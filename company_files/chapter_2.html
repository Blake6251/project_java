백엔드 관점 요약

  • 망·서버·장비 조건을 먼저 분석해 배포 형태를 결정.
  • 실시간 흐름과 요청·응답 API를 분리해 장애 지점과 책임 범위를 격리.
  • 인증·다운로드·기기 검증을 단계별로 나누어 단일 지점 우회를 차단.
  • 운영 중 확인된 문제를 다음 프로젝트 초기 구조에 반영해 반복 제거.
  • 요구사항이 불명확하면 기능·구조·운영 범위를 직접 정의하고 납품까지 책임.

프로젝트

1. MSDS 키오스크 — KT&G 외 5개 고객사 (2024.08~현재)

→ 공장 현장용 화학물질 조회·관리 키오스크를 5개 고객사에 납품.

→ 제약: 고객사마다 서버 유무·망·보안 정책이 달라 단일 배포로 대응 불가.

→ 해결: 동일 코드 베이스에서 분리형 / 단일 풀스택 / 컨테이너 3가지 배포 구조로 분기 설계.

→ 판단: 단일 배포 전략 대신 분리형 구조를 선택. 배포 복잡도는 상승했지만 고객사별 환경 대응 유연성을 확보하는 트레이드오프를 수용.

→ 성능: 주요 조회 컬럼 인덱스 재설계 및 N+1 쿼리 제거로 현장 검색 응답 시간 안정화.

→ 결과: 제품 하나로 5개 현장 제약 흡수, 신규 고객사 대응 기간 단축.

2. 안전교육 퀴즈 키오스크 — 포스코 E&C, 수협중앙회 외 (2025.04~2025.11)

→ 안전교육 이수 평가용 퀴즈 키오스크 공통 솔루션을 고객사별로 커스터마이징 납품.

→ 제약: 체험 장비 연동 버전에서 UI 반응과 기계 타이밍이 뒤엉켜 동작 불안정.

→ 해결: UI·업무 규칙과 장비 제어를 계층 분리, 상태 변화는 이벤트 단방향 전달.

→ 결과: 장비 연동 버전 동작 신뢰도 안정화, 내부망에서 결과 통지·상태 전송 운영 흐름 확보.

→ 운영: 평균 10~20대 단말 동시 운영 환경에서 안정 동작 검증, 현장 장애 빈도 감소.

3. 시뮬레이터 / 원격 제어 — 배달의민족 체험안전센터 (2026.01~현재)

→ VR 시뮬레이터-제어기 실시간 연동, 센터 내 다수 PC 원격 전원·프로그램 제어 시스템 구축.

→ 제약: 상태 전송과 조회·저장 요청이 한 경로에 섞이면 지연·유실 위험.

→ 초기 시도: 단일 이벤트 흐름으로 설계해 상태 꼬임·메시지 순서 문제 발생 → 채널 분리 구조로 재설계.

→ 해결: 상태 채널(WebSocket)과 요청 채널(HTTP API)을 분리, heartbeat·재연결 로직으로 실시간 통신 안정성 확보.

→ 결과: 현장 전 구간 통합 테스트 통과 후 납품, 운영 중 상태 꼬임 이슈 없음. 장애 발생 시 평균 복구 10분 이내 대응.
